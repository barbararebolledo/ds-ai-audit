# AI-Ready Design System — Infrastructure Guide

**Project:** AI-Ready Design System POC  
**Context:** Accenture Song, Nordic Design Director led project  
**Purpose:** Reference document for design system decisions, token architecture, naming conventions, and the vision for AI-integrated design systems.  
**Use this document as a source when asking Claude questions about this project.**

---

## What we are building and why

We are building a design system where both humans and machines can read, understand, and act on the system's components, tokens, and patterns without ambiguity.

The core idea: an AI agent needs the same information as a human designer or developer to use the design system correctly. If that information is not structured, named, and organised in a way machines can parse, the AI either guesses or fails. The goal is to eliminate guessing.

The practical outcome: Claude Code can read a Figma file, understand a component completely, generate a structured contract describing that component, and produce documentation from that contract -- automatically, without human intervention, and without making things up.

This is not about replacing designers. It is about augmenting them. The AI handles the repetitive, structural work -- documentation generation, token auditing, consistency checking. The designer handles intent, judgement, and the decisions that require human context.

---

## The infrastructure that must exist

A mature AI-ready design system requires infrastructure at three levels: Figma, the repository, and the connection between them.

### In Figma

**Figma Variables with a three-layer token architecture**  
All design decisions must be stored as Figma Variables, not as Styles, not as hardcoded values. Variables support alias chains and modes -- the two capabilities that make the system machine-readable and scalable. See the full explanation in the Variables vs Styles section below.

**All component properties bound to Variables**  
Every fill, stroke, radius, padding, gap, and typography property on every component must be bound to a component-level Variable. Not a semantic Variable, not a primitive, not a hardcoded value. A component-level Variable. This is non-negotiable. If a property is not bound to a Variable, the AI cannot read it as part of the token chain.

**Component descriptions written for every component**  
The description field on a Figma component is read by the official Figma MCP and passed directly to Claude Code. It becomes the first line of the component contract. If it is empty, Claude has to infer the component's purpose from its name alone. Write descriptions that explain what the component is for, when to use it, and when not to.

**Consistent naming conventions throughout**  
See the full naming conventions section below. The short version: use full words, use slashes as separators, name by role not by appearance.

### In the repository

**CLAUDE.md at the root**  
A plain-language governance file that tells Claude Code what the system is, how to navigate the repo, what it is allowed to do, and what it is not allowed to do. Claude Code reads this file first, every session. If it is inaccurate or missing, every session starts from a broken state.

**A component index**  
A machine-readable JSON file listing every component that exists, where its contract lives, and what its Figma node ID is. A human-readable Markdown version alongside it. The index includes a `notYetDefined` array -- an explicit list of components that do not exist yet. This prevents Claude Code from inventing components by inferring from context.

**Component contracts**  
One JSON file per component. The contract is the source of truth for everything about that component: identity, intent, props, token map, when to use, when not to use, constraints, alternatives. The contract is generated from Figma, not written by hand. See the contract section below.

**Token files mirroring Figma Variables**  
JSON files that reflect the three layers of Variables in Figma. Generated by reading the Figma REST API. These files give Claude Code the full token chain without having to make Figma API calls every time it needs to resolve a value.

**Documentation generated from contracts**  
Markdown files for human consumption. Generated from the contract, never written independently. If the contract updates, the documentation is regenerated. One source of truth, two outputs.

### The connection

**Official Figma MCP server**  
Enabled in Figma desktop under Preferences. Connects Claude Code directly to the Figma file. Used for reading component structure, design context, and variable definitions.

**Figma REST API**  
Used for reading raw Variable data with full alias chains intact. The MCP returns resolved values only -- the REST API returns the alias structure. Required for generating accurate token files.

**Claude Code**  
The AI agent that reads from Figma, generates contracts, generates documentation, writes to the repo. Not Claude.ai (the chat interface). Claude Code is a command line tool that operates inside a project directory.

## Token architecture: the three layers

### Why three layers and not one

A single layer of tokens -- just a list of values -- gives you no ability to change things systematically. Three layers give you three levels of control:

Change a primitive: everything that aliases through it updates automatically.  
Change a semantic token: everything that aliases through it updates automatically.  
Change a component token: only that component updates.

This means a rebrand -- changing the primary blue to a different blue -- is one value change in the primitives layer. A theme change -- switching from a blue primary to a green primary -- is one alias change in the semantic layer. A component-specific adjustment -- making a specific button a slightly different shade -- is one change in the component layer.

Without three layers, any of these changes requires hunting through the system manually and updating every instance. With three layers, each type of change has exactly one place to make it.

### Layer 1: Primitives

**What they are:** Raw values. The complete palette and scale of the system. No meaning, no context.

**Named by:** What they are, not what they do. Scale position for colours, numeric steps for spacing and radius.

```
color/blue/600      #2563EB
color/neutral/0     #FFFFFF
color/red/600       #DC2626
spacing/4           16
spacing/6           24
radius/md           8
typography/size/md  16
```

**Rules:**
- No aliases at this layer. Only raw values.
- No functional grouping. `color/red/600` not `color/feedback/error`. The red does not know it will be used for errors. That meaning gets assigned at the semantic layer.
- Every colour that exists in the system lives here. If it is not here, it should not exist in the system.

**Why no functional grouping at primitives:**  
If you call it `color/feedback/error` at the primitive layer, you cannot later use that same red for a destructive action button without the name lying about its purpose. Primitives are neutral. Meaning is a semantic concern.

### Layer 2: Semantic

**What they are:** Aliases to primitives. They give meaning to raw values.

**Named by:** Role and intent. What is this value for, not what does it look like.

```
color/action/primary          → primitives/color/blue/600
color/action/primary-hover    → primitives/color/blue/700
color/text/on-action          → primitives/color/neutral/0
color/feedback/error          → primitives/color/red/600
spacing/component-md          → primitives/spacing/4
```

**Rules:**
- No raw values. Every semantic token is an alias to a primitive.
- Named by role, never by appearance. `color/action/primary` not `color/blue/brand`.
- This is where theming happens. To switch the primary action colour from blue to green, you change one alias: `color/action/primary` points to `color/green/600` instead of `color/blue/600`. Every component that uses that semantic token updates automatically.

**Why the semantic layer exists beyond just making changes easier:**  
It also makes the system legible to AI. When Claude Code sees `color/action/primary`, it can reason: this is the colour for the primary interactive action. It does not need to be told. The name carries the meaning. Without the semantic layer, Claude Code sees `color/blue/600` applied to a button background and has to guess at the intent.

### Layer 3: Component

**What they are:** Aliases to semantic tokens, scoped to a specific component.

**Named by:** Component name, then property type, then property state.

```
button/color/background           → semantic/color/action/primary
button/color/background-hover     → semantic/color/action/primary-hover
button/color/background-disabled  → semantic/color/action/primary-disabled
button/color/label                → semantic/color/text/on-action
button/spacing/padding-x-md       → semantic/spacing/component-md
button/radius/md                  → semantic/radius/component-md
```

**Rules:**
- No raw values. No direct aliases to primitives. Only aliases to semantic tokens.
- Named with component prefix so the scope is unambiguous.
- Structure is always: component / property-type / property-state.

**Why the component layer exists beyond just scoping:**  
Components sometimes need to deviate from the semantic value. A ghost button and a primary button both express a primary action, but their backgrounds are different. The component layer gives each its own token without touching the semantic layer. It also makes the AI contract precise -- `button/color/background-hover` tells Claude Code exactly which visual property changes on hover, unambiguously.

### The alias chain rule

Component tokens alias semantic tokens.  
Semantic tokens alias primitives.  
No shortcuts.

If a component token aliases a primitive directly, the semantic layer is bypassed. The system loses the ability to retheme at the semantic level. The AI contract loses the ability to identify the role of that token. This rule is an architectural invariant -- it must never be broken.

## Token structure at scale: multi-theme and multi-platform

### Single theme, single platform (current POC)

```
Collection: primitives    (raw values, no modes needed)
Collection: semantic      (aliases to primitives, one mode)
Collection: component     (aliases to semantic, one mode)
```

### Adding dark mode

Add a second mode to the semantic and component collections. Primitives do not change per theme -- you do not need new primitive values for dark mode, you just alias to different existing primitives.

```
Collection: primitives    (no modes, raw values stay the same)
Collection: semantic      modes: Light / Dark
Collection: component     modes: Light / Dark
```

In Light mode, `color/action/primary` points to `color/blue/600`.  
In Dark mode, `color/action/primary` might point to `color/blue/400` -- a lighter blue that works on dark backgrounds.

The component tokens do not need to change. They still alias `color/action/primary`. The mode handles the rest automatically.

### Adding multiple platforms (mobile, tablet, web)

Figma Variables support one mode dimension per collection. You cannot have a native matrix of theme x platform in one collection. The recommended solution is two separate semantic collections:

```
Collection: primitives          (no modes)
Collection: semantic-theme      modes: Light / Dark
Collection: semantic-platform   modes: Mobile / Tablet / Web
Collection: component           modes: Light / Dark (inherits from semantic-theme)
```

**Why split semantic into two collections:**  
Colour responds to theme. Spacing, typography, and radius respond to platform. These are different concerns and separating them reflects that reality. A spacing token does not need a Light and Dark value. A colour token does not need a Mobile and Web value. Combining them into one collection with six modes (Light-Mobile, Dark-Mobile, Light-Tablet, Dark-Tablet, Light-Web, Dark-Web) works technically but becomes unmanageable quickly.

**What changes per platform in the semantic layer:**
```
semantic-platform
  spacing/component-md
    Mobile:  12px
    Tablet:  16px
    Web:     20px
  
  typography/label-md/size
    Mobile:  14px
    Tablet:  16px
    Web:     16px
```

**What changes per theme in the semantic layer:**
```
semantic-theme
  color/action/primary
    Light: → primitives/color/blue/600
    Dark:  → primitives/color/blue/400
  
  color/surface/page
    Light: → primitives/color/neutral/50
    Dark:  → primitives/color/neutral/900
```

Components bind to both collections. The platform mode and theme mode are applied independently.

### In Figma collections: one component collection or one per component?

One collection for all component tokens in Figma. The grouping within that collection (using slash prefixes like `button/`, `input/`, `card/`) provides the separation.

```
Collection: component
  button/color/background
  button/spacing/padding-x-md
  input/color/border
  input/spacing/padding-x-md
  card/color/background
```

In the repository, the extraction script splits this into separate files by component prefix:
```
tokens/component/button.json
tokens/component/input.json
tokens/component/card.json
```

The split happens in code, not in Figma.

## Naming conventions

### Why naming conventions matter

Naming conventions are not aesthetic preferences. They are information architecture. The name of a token is the only thing Claude Code has to understand what that token is for, how it relates to other tokens, and where it belongs in the system.

A human reading `bg-h` might guess it means background-hover. Claude Code makes the same guess -- but a guess is not a certainty. Across a system with hundreds of tokens, small uncertainties accumulate into contract errors. A wrong interpretation in a token name propagates silently through the contract, into the documentation, and into anything generated from it.

The standard is: name tokens as if the reader has zero context about your system and cannot ask questions.

### The four naming decisions

**Slash as separator**

Use: `color/blue/600`  
Not: `color-blue-600` or `colorBlue600`

Figma uses the slash to create group hierarchy in the Variables panel. The Figma MCP returns token paths using slashes. Claude Code can parse `color/blue/600` as category, subcategory, scale step. Hyphens produce one long unparseable string. CamelCase loses the hierarchy entirely.

In code, Style Dictionary transforms slashes to the appropriate format:
- CSS: `--color-blue-600`
- JS/TS: `tokens.color.blue[600]`

**Numeric scales for primitives**

Use: `blue/600`  
Not: `blue/dark` or `blue/primary`

Numbers are honest. `blue/dark` is already an interpretation -- dark compared to what? Numbers let you insert values above and below without renaming. They match the Tailwind scale convention which most developers already understand.

**Role-based names for semantic tokens**

Use: `color/action/primary`  
Not: `color/blue/brand` or `color/button-background`

Semantic tokens describe intent, not appearance or location. `color/action/primary` tells Claude Code what the colour is for without describing what it looks like. Claude Code can reason: this is a primary action, therefore this is the right token for a CTA button. It does not need to be told explicitly.

`color/blue/brand` requires knowing your brand colour is blue. `color/button-background` only works for buttons.

**Property-and-state for component tokens**

Use: `button/color/background-hover`  
Not: `button/hover-background` or `button/bg-h`

Structure is always: component / property-type / property-state. Predictable. If you know a component has a `background` token, you can infer it also has `background-hover`, `background-active`, `background-disabled` without reading every token individually. Claude Code does the same inference.

### On abbreviations

`bg` is widely understood in the industry. A developer will not be confused by it. The risk is not comprehension -- it is consistency over time and across people.

Once you allow abbreviations, the boundary becomes arbitrary. Is it `bg` or `background`? `txt` or `text`? `col` or `color`? Different people make different calls. A year later the system has `button/color/bg-hover` and `card/color/text-disabled` and `input/col/border-active`. The pattern is inconsistent and Claude Code cannot reliably parse the structure.

Full words are not about whether `bg` is understood. They are about having one unambiguous rule that leaves no room for drift.

### Naming conventions in code versus Figma

The slash convention is Figma-specific. In code:
- CSS custom properties: slashes become hyphens (`--button-color-background-hover`)
- JS/TS objects: slashes become nested keys
- Style Dictionary handles this transformation automatically

If you set Code Syntax in Figma Variables -- there is a field for it in the variable settings -- you can specify the exact code variable name. The Figma MCP reads this field and passes it to Claude Code, so there is no ambiguity about how the Figma token maps to code.

## Variables versus Styles

### Why Variables and not Styles

Styles are Figma's older system for storing reusable design properties. Most design systems built before 2023 use Styles. They work well for simple use cases but have three fundamental limitations that make them insufficient for an AI-readable system.

**Alias chains**  
Styles store raw hex values. A colour Style cannot point to another Style. The entire three-layer token architecture requires aliasing -- primitives to semantic, semantic to component. Styles cannot do it. Without aliasing, every Style is an island. Changing a brand colour means updating every Style that uses it manually.

**Modes**  
Dark mode, platform variants, and theming require the same token to resolve differently per context. Figma Variables have modes built in -- you define Light and Dark as modes on a collection and values switch automatically. Styles have no equivalent mechanism. Every theme variant requires duplicating the entire Style library.

**MCP readability**  
The official Figma MCP reads Variable bindings via `get_variable_defs`. It returns the full alias chain. It does not traverse Style-to-Variable relationships in the same way. A component bound to Styles produces incomplete MCP output -- Claude Code sees Style names but cannot trace them back through a token chain. A component bound directly to Variables produces a clean, fully traceable contract.

### What about Style-with-Variable-alias?

You can bind a Style's value to a Variable -- so the Style's colour is driven by a Variable rather than a hardcoded hex. This is better than a raw Style. It gives you mode support. The value updates when the Variable changes.

But the MCP still sees the Style name on the component layer, not the Variable binding underneath it. Claude Code cannot traverse the Style-to-Variable relationship the way it can traverse Variable-to-Variable aliases. In an AI-readiness audit, Style-with-Variable-alias scores better than raw Styles but lower than direct Variable bindings.

For a system that needs to be fully AI-readable, direct Variable bindings on components is the correct architecture.

### Why every component property must be a Variable

When Claude Code reads a component via the Figma MCP, it sees the properties of the component and their values. If a property is bound to a Variable, Claude Code gets the Variable name, which it can trace through the token chain. If a property has a hardcoded value, Claude Code gets a hex code or a number with no context -- it cannot tell what role that value plays, what it relates to, or whether it is intentional.

The component contract maps every property to a token. That map is only possible if every property has a token to map to. One hardcoded value breaks the completeness of the contract. In a system where contracts feed AI generation, an incomplete contract produces incomplete output.

This applies to every property: fill, stroke, corner radius, padding (top, right, bottom, left), gap, font family, font size, font weight, line height. Every property. No exceptions.

## What the system produces

### Component contract (machine-readable)

A JSON file generated by Claude Code after reading the Figma component. It contains:

- **Identity:** component name, category, version, Figma file key, Figma node ID
- **Intent:** what this component is for in plain language, drawn from the component description in Figma
- **Props:** all variant properties with types, values, and defaults
- **Tokens:** complete map of which token controls which property, per variant where relevant
- **When to use:** explicit rules for correct usage
- **When not to use:** explicit rules for incorrect usage
- **Constraints:** what cannot be changed or overridden
- **Alternatives:** what to use if this component does not fit

The contract is the source of truth. Everything else is derived from it.

### Documentation (human-readable)

A Markdown file generated from the contract. Contains the same information presented for human consumption -- props tables, state tables, token maps with resolved values, alias chain examples, usage guidance.

The documentation is never written independently. It is always generated from the contract. If the contract updates, the documentation is regenerated. This is what eliminates documentation drift -- the problem where docs fall out of sync with the component because they are maintained separately.

### Component index

A JSON file listing every component in the system: name, status, Figma coordinates, contract path, documentation path. Includes a `notYetDefined` array of components that do not exist yet. This prevents Claude Code from inventing components that have not been built.

A human-readable Markdown version alongside the JSON.

## The vision: what this system enables

### Now (POC validated)

Claude Code reads a Figma component, generates a contract, generates documentation. Automated, consistent, accurate. No manual documentation writing. No documentation drift.

### Near term

**Token audit**  
Point Claude Code at any existing design system Figma file. It reads the Variables, checks alias chain integrity, evaluates naming conventions, verifies component bindings, scores the system on AI-readiness. Produces a report showing exactly what needs to change and why. Valuable as a client deliverable.

**Dark mode generation**  
Given the light mode token system, Claude Code proposes dark mode values with reasoning -- which colours invert, which stay the same, which need adjustment for contrast. Written back to Figma via the REST API.

**Component generation**  
Given a contract and a prompt, Claude Code generates a component implementation in React, Swift, Kotlin, or any target framework. The component uses the correct tokens, respects the constraints defined in the contract, and matches the Figma specification.

### Medium term

**Figma plugin for AI-readiness auditing**  
A plugin UI inside Figma showing AI-readiness scores per component, flagging broken alias chains, highlighting missing descriptions, identifying hardcoded values. The data comes from the audit JSON files in the repo. The plugin renders them visually without requiring Claude Code to be running.

**Documentation site**  
A frontend (static site or Next.js) that renders the Markdown documentation files from the repo. When a contract regenerates and produces new documentation, the site updates automatically. Designers and developers see always-current documentation without anyone maintaining it manually.

**Screen generation**  
Given a prompt and the component index, Claude Code assembles a screen from existing components following the composition rules in the DS Composer layer. Validates the output against the system before delivering it.

### Long term: GenUI

The design system becomes the constraint layer for AI-generated UI. Any AI tool that generates screens -- Figma Make, similar tools -- operates within the rules and components defined in the system. The AI does not invent components, does not use colours outside the token system, does not violate composition rules. The design system is the governance layer for AI-generated interfaces.

This is the direction the industry is moving. Design systems built today without this infrastructure will need to be rebuilt to participate in that workflow. Design systems built with this infrastructure are already ready.

## What must never change

These are architectural invariants. They apply at POC scale and at production scale.

**The alias chain**  
Component tokens alias semantic tokens. Semantic tokens alias primitives. No shortcuts. Breaking this breaks the AI's ability to reason about the system.

**The slash naming convention**  
Inconsistent separators prevent Claude Code from parsing the token structure. One convention, applied everywhere, always.

**Variable bindings on every component property**  
If any property has a hardcoded value, the contract is incomplete. Incomplete contracts produce incomplete outputs.

**Contract as source of truth**  
Documentation is always derived from the contract. Editing documentation directly creates drift. The generation direction -- contract first, documentation second -- is a governance rule.

**CLAUDE.md as the entry point**  
Claude Code reads this first, every session. It must stay accurate as the repo evolves.

## Glossary

**Alias:** A Variable that points to another Variable rather than storing a raw value. The chain of aliases from component to semantic to primitive is what makes the token architecture work.

**Claude Code:** A command line AI agent that operates inside a project directory. Not the same as Claude.ai (the chat interface). Claude Code reads and writes files, calls APIs, and connects to MCP servers.

**Component contract:** A JSON file that captures everything an AI agent needs to know about a component -- identity, intent, props, token map, usage rules.

**Figma MCP:** The official Figma Model Context Protocol server. Connects Claude Code to a Figma file for reading design context, component data, and variable definitions.

**Figma REST API:** Figma's HTTP API. Used for reading raw Variable data with full alias chains. Also supports writing Variables -- used for the dark mode POC.

**Intelligence layer:** The set of files in the repo that make the design system legible to AI -- CLAUDE.md, manifest, token files, contracts, index, documentation. No component code. The component lives in Figma.

**Mode:** A Figma Variable setting that allows the same token to resolve to different values per context. Used for Light/Dark theming and Mobile/Tablet/Web platform variants.

**Primitive:** The first layer of the token architecture. Raw values named by scale position. No aliases, no meaning.

**Semantic token:** The second layer. An alias to a primitive, named by role and intent.

**Component token:** The third layer. An alias to a semantic token, scoped to a specific component and named by property and state.

**Style Dictionary:** An open source tool that transforms token JSON files into platform-specific outputs -- CSS custom properties, Swift constants, XML for Android. Handles the conversion from Figma's slash naming to code conventions.
